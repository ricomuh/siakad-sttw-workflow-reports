## E2E Test Results

E2E Skipped — Reason: Project configuration mismatch.

`admin-blended.spec.ts` belongs to project `siska-admin` (covers all siska specs except `admin-kemahasiswaan.spec.ts`), but the spec uses multiple `test.use({ storageState })` overrides per `describe` block (mahasiswa, dosen, kaprodi) while `siska-admin` is configured with a single waket1 storageState and `dependencies: ['auth-setup']`.

Running `npx playwright test tests/e2e/siska/admin-blended.spec.ts --project=siska-admin --reporter=list` only executed auth-setup dependency tests (5 auth setups: admin, ketua, waket1, waket2, waket3) — the spec's own test blocks did not run.

Auth state files required by the spec (`.auth/mahasiswa.json`, `.auth/dosen.json`, `.auth/kaprodi.json`) exist in the auth setup pipeline but the multi-role nature of the spec requires those sessions to be active. The spec is structured for cross-role testing (3 describe blocks × different roles) which is the correct design — execution requires the full multi-role auth setup to be complete.

## Factory Fixes

PR #515 includes the following factory fixes:

1. **BlendedLogbookFactory** — outdated fields removed and aligned with current `blended_logbooks` schema. Fields like `kegiatan` → `aktivitas`, added `status` ('pending'|'disetujui'|'ditolak'), `komentar_dosen`, `file_path`.
2. **BlendedPeriodeFactory** — added `bobot_dosen` and `bobot_perusahaan` fields (used in `calculateFinalNilai()` formula). Without these fields, weighted average tests would fail with NULL values.
3. **Dosen/BlendedLogbookController** — relation name bug fixed. Incorrect Eloquent relation call was causing a runtime error when dosen accessed logbook bimbingan list.

## Pest Test Detail

### BlendedLogbookMahasiswaTest (17 tests)

Key scenarios verified:
- `mahasiswa can view own logbooks index` — asserts 200 + sees aktivitas text
- `mahasiswa without active blended gets 403 on index` — status set to 'selesai' → assertForbidden
- `mahasiswa can open create logbook form` — assertOk
- `mahasiswa can create logbook with tanggal and aktivitas` — POST → assertRedirect + assertDatabaseHas
- `mahasiswa cannot create when status is not aktif` — assertForbidden
- `mahasiswa can upload file in logbook` — Storage::fake('public'), UploadedFile fake
- `mahasiswa cannot upload invalid file type` — validation error on non-pdf/doc
- `mahasiswa can edit own pending logbook` — PUT → assertRedirect
- `mahasiswa cannot edit disetujui logbook` — assertRedirect + assertSessionHas('error')
- `mahasiswa can delete own pending logbook` — DELETE → assertDatabaseMissing
- `mahasiswa cannot delete disetujui logbook` — assertSessionHas('error') + assertDatabaseHas
- `mahasiswa cannot delete logbook belonging to another student` — assertForbidden

### BlendedDosenLogbookTest (9 tests)

Key scenarios verified:
- `dosen can view list of bimbingan mahasiswa` — assertOk + assertSee mahasiswa nama
- `dosen only sees their own bimbingan mahasiswa` — isolation: otherMahasiswa NOT visible
- `dosen can view logbook detail of their mahasiswa`
- `dosen cannot view logbook of other dosen's mahasiswa` — assertForbidden
- `dosen can add komentar to logbook` — POST komentar → assertRedirect
- `dosen cannot approve already-disetujui logbook` — guard check
- `dosen can setujui logbook` — status → 'disetujui'
- `dosen can tolak logbook` — status → 'ditolak' + komentar required

### BlendedPenilaianPerusahaanTest (13 tests)

Key scenarios verified:
- SkalaHuruf mapping: nilai_bobot ≥ 85 → 'A', 80–84 → 'AB', 75–79 → 'B', dst.
- `calculateFinalNilai` formula: `(nilai_angka × bobot_dosen × 0.01) + (nilai_perusahaan_angka × bobot_perusahaan × 0.01)`
- Boundary: bobot_dosen=50, bobot_perusahaan=50 → `(90×50×0.01) + (80×50×0.01) = 85.0`
- Boundary: bobot_dosen=60, bobot_perusahaan=40 → `(75×60×0.01) + (65×40×0.01) = 71.0`
- `calculateFinalNilai persists the result to database` — assertDatabaseHas nilai_bobot

### BlendedPresensiBlockTest (11 tests)

Key scenarios verified:
- `isBlended` flag on BlendedMahasiswa triggers presensi skip logic
- Mahasiswa with aktif blended status is skipped in presensi rekap
- Weighted average computation accounts for blended periode correctly
- Mahasiswa selesai/non-aktif status does NOT block presensi
